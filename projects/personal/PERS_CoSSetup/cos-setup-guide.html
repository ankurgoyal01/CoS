1 What is Chief of Stuff?

Chief of Stuff (CoS) is a personal AI-powered command center. You set it up once on your computer, and then you have an assistant that can read your emails, manage your tasks, write documents, analyze data, review architecture, and automate any workflow you can describe in plain language.

It's built on Claude Code — Anthropic's AI agent that runs in your terminal. But instead of using Claude as a blank slate every time, CoS gives it structure: instructions on how to work, memory that persists across sessions, reusable workflow templates, and connections to all the services you use.

Think of it this way: Claude Code is the engine. CoS is the car you build around it — steering wheel, navigation, cargo space, and all.

The Core Idea: English Is Your Programming Language
Throughout this guide, you'll notice something: most steps are just telling Claude what to do in plain English. You don't need to memorize terminal commands, file paths, or configuration syntax. You describe what you want, and Claude does it.

The only things you need to do manually are: install 4 programs (Node.js, Git, VS Code, Claude Code) and copy one settings file from the reference zip. Everything else — creating folders, writing files, installing plugins, connecting services — you just ask Claude.

What can you do with it?

Management & Executive Work
"Process my unread Asana notification emails, categorize tasks by urgency, and create an HTML dashboard." — Claude reads 37 emails, extracts task IDs, fetches details from Asana, and delivers a color-coded interactive report. In about 5 minutes, no manual sorting.
Technical Leadership
"Review our QA documentation against modern best practices and rate each document." — Claude reads 8 internal docs, analyzes them, sends them to 3 different AI reviewers for independent feedback, and delivers a structured review with ratings, strengths, weaknesses, and an action plan.
General Productivity
"Research our competitor's pricing strategy and draft a comparison analysis." — Claude searches the web, reads documents, synthesizes findings, and produces a structured report. You refine it in conversation: "Make the executive summary shorter. Add a risk section."

2 Glossary

You'll encounter these terms throughout the guide. All explained in plain English — no prior knowledge needed.

Terminal / Command Line
A text-based interface on your computer where you type commands instead of clicking buttons. On Mac it's called "Terminal", on Windows it's "Command Prompt" or "PowerShell".
This is where you'll run Claude Code and type your instructions.
Git
A tool that tracks every change you make to your files — like an infinite undo button with bookmarks. When you "commit," you save a snapshot you can always go back to.
CoS uses Git to safely track your work. If anything goes wrong, you can always revert.
Repository (repo)
A project folder that's being tracked by Git. Every project in CoS is its own repository.
Each of your projects gets its own repo so changes in one project don't affect another.
CLI
Command Line Interface — a program you interact with by typing commands (as opposed to clicking buttons in a graphical app).
Claude Code, Codex, and Gemini are all CLIs. You type, they respond.
Node.js / npm
Node.js is a program that runs JavaScript on your computer. npm (Node Package Manager) is its app store — it lets you install tools like Claude Code.
You need Node.js installed so you can install Claude Code via npm.
VS Code
Visual Studio Code — a free code editor made by Microsoft. Think of it as a smart text editor with a built-in terminal and file browser.
Your main workspace. You'll see your files on the left, edit documents in the center, and talk to Claude in the bottom terminal.
CLAUDE.md
A special file that tells Claude how to behave in your workspace. It contains your preferences, rules, and conventions. Claude reads this file at the start of every session.
This is the brain of your CoS — it's how Claude knows your style, your projects, and your expectations.
MCP Server
Model Context Protocol — a bridge that connects Claude to an external service. For example, an MCP server for Gmail lets Claude read your emails. One for Asana lets Claude manage your tasks.
Without MCP servers, Claude can only work with files on your computer. With them, it can reach out to Gmail, Jira, Asana, and more.
Plugin
An add-on that gives Claude new capabilities — like browser automation, code review tools, or connections to specific services.
Plugins extend what Claude can do. Install only the ones you need.
Skill
A reusable workflow template that you create for Claude. It's a file with instructions that Claude follows when you invoke the skill — like a recipe for a specific task.
Skills let you automate recurring workflows. Create one for "weekly email digest" and Claude follows the same process every week.
Hook
A script that runs automatically when something happens — for example, every time you start a new Claude session, or every time you type a prompt. Hooks can inject reminders, log actions, or enforce workflows.
Hooks automate things Claude would otherwise forget. They're the enforcement layer — instructions written as prose get skipped; hooks that fire on every action don't.
Rule
A file in .claude/rules/ that gets injected into Claude's system prompt on every message. Unlike CLAUDE.md instructions (which can get lost in long files), rules are always visible to Claude.
Rules enforce behavior more reliably than prose instructions. Use them for things Claude must never skip — like invoking the right skill before acting.
Subagent
A separate Claude worker that gets spawned to handle a specific subtask. The main Claude stays focused on the big picture while subagents handle research, analysis, or parallel work.
Subagents let Claude do multiple things at once — like researching three topics simultaneously.
Second Opinion
Sending your work (a plan, analysis, or document) to other AI models for independent review. They point out blind spots, challenge assumptions, and suggest improvements.
Different AI models think differently. Getting multiple perspectives catches mistakes that one model alone would miss.

Go to nodejs.org and download the LTS (Long Term Support) version. Open the downloaded file and follow the installer. Alternatively, if you have Homebrew:

brew install node

Verify it worked by opening Terminal and typing:

node --version

You should see something like v22.x.x. Any version 20 or higher is fine.

Go to nodejs.org and download the LTS version (.msi installer). Run the installer and accept all defaults.

Verify by opening PowerShell or Command Prompt and typing:

node --version

You should see something like v22.x.x. Any version 20 or higher is fine.

2. Install Git

Open Terminal and type:

git --version

If Git isn't installed, macOS will pop up a dialog asking to install "Xcode Command Line Tools." Click Install and wait (a few minutes). Then try git --version again — you should see a version number.

Download Git for Windows from gitforwindows.org. Run the installer — accept all the default settings (there are many screens, just click Next).

Verify by opening a new PowerShell window:

git --version

3. Install VS Code

Download from code.visualstudio.com. Install it like any other application. Open it when done — you'll see a welcome screen.

4. Install Claude Code

Open a terminal (on Mac: Terminal app; on Windows: PowerShell; or the terminal inside VS Code) and run:

npm install -g @anthropic-ai/claude-code

This downloads and installs Claude Code globally on your computer. Once it's done, sign in:

claude

Claude will open and prompt you to authenticate with your Anthropic account. Follow the instructions — it will open a browser window where you log in. Once authenticated, you're in.

Verify the installation:

claude --version
Checkpoint: You should now have Node.js, Git, VS Code, and Claude Code installed. If any of these steps failed, ask Claude itself for help — just type your error message and it will guide you.

5. Set Up YOLO Mode (recommended)

By default, Claude asks permission before every action — creating files, running commands, editing things. This is safe but slow. Since you'll be working with Claude constantly, you want it to just do things without asking every time.

YOLO mode (officially called --dangerously-skip-permissions) tells Claude to act without asking. Create a short alias so you don't have to type the long flag every time:

Open your terminal config file and add this line:

echo 'alias yolo="claude --dangerously-skip-permissions"' >> ~/.zshrc && source ~/.zshrc

Now you can start Claude in YOLO mode by just typing:

yolo

In PowerShell, create a profile function:

if (!(Test-Path $PROFILE)) { New-Item $PROFILE -Force } Add-Content $PROFILE 'function yolo { claude --dangerously-skip-permissions @args }'

Restart PowerShell, then start Claude in YOLO mode by typing:

yolo
Why is it safe? In the next step, you'll copy a settings.local.json file from the reference zip that defines exactly which permissions Claude gets. This acts as your security guardrails — Claude can read files and run safe commands freely, but destructive operations still require your approval. YOLO mode + settings file = fast AND safe.

6. Copy Security Settings from Reference Zip

The reference zip includes a pre-configured permissions file that controls what Claude can do without asking. This is your safety net in YOLO mode.

  1. Unzip the reference materials
  2. Find the file at .claude/settings.local.json inside the zip
  3. Copy it to your Claude config folder:
cp path/to/reference/.claude/settings.local.json ~/.claude/settings.local.json
copy path\to\reference\.claude\settings.local.json %USERPROFILE%\.claude\settings.local.json

This file pre-approves safe operations (reading files, git commands, searching) while keeping dangerous ones gated. You can review and customize it later — it's just a JSON file.

Checkpoint: Now when you type yolo, Claude starts in YOLO mode with sensible security guardrails. Fast and safe.

7 Set Up Second Opinions

One of the most powerful features of CoS is the ability to get independent AI review of your work. When Claude creates a plan, analysis, or document, you can send it to other AI models for a fresh perspective.

Think of it as a built-in peer review system. Different models think differently — they catch blind spots, challenge assumptions, and improve quality.

Option A: Claude Only (no extra setup)

Simplest — already available
The second-opinion skill can spawn a separate Claude process to review work. No extra accounts or API keys needed.

Limitation: It's the same model reviewing itself. Better than no review, but it shares the same blind spots.

When this is enough: Quick reviews, non-critical work, getting started.

Option B: Multi-Model Review (recommended for important work)

For the best results, add two more AI reviewers so you get three independent perspectives:

1. Claude — already available

Comes with your Claude subscription. No extra setup needed.

2. OpenAI Codex CLI

OpenAI's coding agent that runs in the terminal.

npm install -g @openai/codex

Authentication (pick one):

  • ChatGPT login (easiest): Run codex, select "Sign in with ChatGPT." Needs a ChatGPT Plus or Pro subscription.
  • API key: Create one at platform.openai.com, then set it in your terminal profile:

Add to your ~/.zshrc file:

export OPENAI_API_KEY="your-key-here"

In PowerShell, set permanently:

[System.Environment]::SetEnvironmentVariable("OPENAI_API_KEY", "your-key-here", "User")

Verify: codex --version

3. Google Gemini CLI

Google's AI agent in the terminal.

npm install -g @google/gemini-cli

Authentication:

  • Get a free API key from aistudio.google.com
  • Set it in your terminal profile (same way as the OpenAI key above, but use GOOGLE_API_KEY)

Verify: gemini --version

How it works in practice

You tell Claude: "Get a second opinion on this plan"

Claude automatically:

  1. Detects which CLIs are installed (Claude, Codex, Gemini)
  2. Sends the work to all available reviewers in parallel
  3. Collects their feedback
  4. Synthesizes: where they agree (high confidence) and where they disagree (needs your judgment)
  5. Presents prioritized action items
Cost note: Codex with ChatGPT login is included in your ChatGPT subscription. Gemini CLI has a generous free tier. So the full three-model stack can be nearly free beyond your existing subscriptions.

Rules — Enforcing Behavior

Rules live in .claude/rules/ and are injected into every Claude message. They're stronger than CLAUDE.md instructions because Claude can't miss them.

The most important rule is skill-enforcement — a decision matrix that maps task types to required skills:

Entry gates (invoke before acting):
Creative/build work → brainstorming skill
Bug/error → systematic-debugging skill
Multi-step task → writing-plans skill

Exit gates (invoke after producing output):
After any plan/analysis/recommendation → second-opinion
After any implementation → ralph-loop
Before claiming "done" → verification-before-completion

Without this rule, Claude optimizes for speed and skips skills. With it, Claude self-checks before every action.

Hooks — The Enforcement Layer

Hooks fire automatically on events. Key hooks in a working CoS setup:

HookFires onWhat it does
cos-welcomeSession startReminds Claude to show the startup menu
skill-nudgeEvery promptPattern-matches your prompt and reminds Claude which skill to invoke (brainstorming, debugging, etc.)
prompt-historyEvery promptLogs every prompt with timestamp to logs/prompt-history.jsonl

Hooks are configured in .claude/settings.local.json under the "hooks" key. The reference zip includes all of these pre-configured.

Prompt Logging — Your Command History

Every prompt you type is automatically logged — no setup needed. A hook fires on every message and saves it to a logs/prompt-history.jsonl file with a UTC timestamp.

CoS-level prompts (when you're in the root workspace) go to logs/prompt-history.jsonl in the CoS root.
Project-level prompts (when you're inside a project) go to logs/prompt-history.jsonl inside that project's folder.

This gives you a full audit trail — you can see what you asked, when, and in which context. Useful for reviewing your workflow, finding past instructions, or understanding how a deliverable was produced.

Project Scaffolding

When you start a new initiative, CoS creates a complete project structure for you. Just say "Create a new project" and Claude will ask for a name, category, and type, then scaffold everything:

projects/groupon/GRPN_CheckoutVisibility/ ├── inputs/ # Drop source materials here │ └── .processed.md # Tracks what's been processed ├── knowledge/ # Extracted & synthesized content │ └── README.md # Navigation map ├── deliverables/ # Final outputs ├── CLAUDE.md # Project context for Claude └── README.md # Project overview
CLAUDE.md — the project brain
Each project gets its own CLAUDE.md tailored to its type (research, analysis, strategy, code, etc.). It contains:
  • Metadata — category, status, creation date, type
  • Purpose — what the project is trying to achieve
  • Workflow — step-by-step process matched to the project type
  • Inputs Processing — rules for auto-ingesting materials from inputs/
  • Domain Learnings — knowledge Claude accumulates as it works on the project
When Claude opens a project, it reads this file first — so it immediately knows the context, goals, and what it learned in previous sessions.
README.md — the human-readable overview
A simple overview of the project: what it is, its status, and the folder structure. Useful when you're browsing projects or sharing context with others.

Projects are registered in docs/projects.md — a central index of everything you're working on. Each gets its own git repo, so you have full version history per project.

Learnings — How CoS Gets Smarter

CoS has two types of memory that persist across sessions:

1. Domain Learnings (per project)
Each project's CLAUDE.md has a ## Domain Learnings section. As Claude works on a project, it appends discoveries: API endpoints, team members, architecture decisions, key facts.

Next time you open that project, Claude already knows what it learned before.
2. Lessons Learned (workspace-wide)
tasks/lessons.md captures mistakes and rules to prevent them. When you correct Claude — "No, always use this format" — it adds a lesson so the same mistake doesn't happen again.

This is how your CoS gets better over time. Every correction makes future sessions smoother.

You can also tell Claude explicitly: "Remember that we always use metric units in this project" — and it adds it to the project CLAUDE.md.

Task Tracking

Simple and practical:

  • tasks/todo.md = a checklist that Claude creates when planning and updates as it works
  • You can say: "Add a task for reviewing the Q2 report" or "What's left to do?"
  • Each project can have its own tasks/todo.md for project-specific tracking

The Review Loop

This is the core workflow that makes CoS outputs reliable:

1. PLAN — tell Claude what you want
→
2. REVIEW — get second opinion on the plan
→
3. EXECUTE — Claude works through step by step
→
4. VERIFY — prove the output is correct
→
5. ITERATE — refine based on results

This loop applies to everything: documents, analysis, research, code. The key habits:

  • Always plan before executing
  • Get second opinions on important work
  • Always verify before declaring done
  • Iterate — first draft is rarely final

▶ Your First Real Task

Time to use your CoS for something real. Pick one of these walkthroughs:

Option A: Create a Project Plan (no services needed)

1. Open VS Code, open the terminal, make sure you're in your CoS folder, and run yolo

2. Type: "Create a new project called 'home renovation' under personal"

→ Claude creates the folder, initializes Git, creates CLAUDE.md

3. Type: "I want to plan a bathroom renovation. Help me create a project plan with phases, budget considerations, and a timeline. Plan this out before starting."

→ Claude enters plan mode, may ask clarifying questions, then presents a structured plan

4. Review the plan. Give feedback: "Add a section about permits. Make the timeline more conservative."

→ Claude revises

5. Type: "Get a second opinion on this plan"

→ Claude sends it for review and presents the feedback

6. Done! Your plan is saved in docs/plans/. Open it in the editor to review.

Option B: Process Your Emails (requires Gmail connected)

1. Open Claude Code in your CoS folder

2. Type: "Search my Gmail for unread emails from this week and summarize the top 10 by urgency"

→ Claude searches Gmail, reads emails, categorizes them

3. Review the summary. Type: "Save this as an HTML report in temp/"

→ Claude creates a styled HTML file

4. Open the file in your browser to see your email digest

5. Type: "Now create a skill so I can run this every Monday"

→ Claude creates a reusable skill in .claude/skills/

Congratulations! You just used your Chief of Stuff. From here, explore what else Claude can do. Ask it anything — process data, write documents, analyze problems, automate workflows. The more you use it, the smarter it gets (through learnings and refined skills).

💡 Tips & Troubleshooting

Tips

  • Start small. Don't try to set up everything at once. Get comfortable with the basics first.
  • Talk naturally. No special syntax needed. Just describe what you want in plain language.
  • Correct Claude. If Claude does something wrong, tell it. It adds a lesson so it doesn't repeat the mistake.
  • Plan before doing. For anything with 3+ steps, ask Claude to plan first.
  • Get second opinions on important work. It takes seconds and catches real issues.
  • Iterate on your CLAUDE.md. This file evolves with you. Add preferences, rules, and conventions as you discover them.
  • Ask Claude to improve itself. "This workflow isn't efficient. How can we make it better?"

Common Issues

ProblemSolution
"Claude says it can't access my email/calendar/tasks"The MCP server isn't connected. Ask Claude: "Help me set up the Gmail MCP server."
"I'm hitting usage limits"If you're on Pro, this is expected — upgrade to Max 5x ($100/mo). It's the plan built for daily CoS use. Pro limits reset every 5-8 hours, but you'll keep hitting them.
"Claude forgot what we discussed"Long conversations get compressed. Put important info in CLAUDE.md or project files — those persist across sessions.
"A plugin isn't working"Ask Claude: "Check if the GitHub plugin is working." It can diagnose and fix most issues.
"git says 'not a repository'"You need to run git init in the folder first. Go back to Step 3.
"npm: command not found"Node.js isn't installed. Go back to Step 1.
"Claude seems slow or stuck"Complex tasks take time. If Claude is truly stuck, type "Stop. Let's rethink this approach."

Getting Help

  • Ask Claude itself: "I'm having trouble with X, help me debug it" — this works surprisingly well
  • Claude Code issues: github.com/anthropics/claude-code/issues
  • The reference repo: Open it as a project and ask Claude to explain any part of it

Cost Management

  • Pro ($20/mo): Good for getting started. Limits reset every 5-8 hours, but you'll hit them fast with real work. Treat it as a trial.
  • Max 5x ($100/mo): The Builder plan. Limits reset weekly with 5x the capacity — enough for daily CoS use. This is where most Builders end up, and it's well worth it once you see what CoS can do.
  • Max 20x ($200/mo): For power users running lots of parallel agents or multiple heavy projects simultaneously.
  • Subagents and parallel work use more quota than simple questions.
  • Second opinions use quota per reviewer (Claude counts, Codex/Gemini use their own quotas).
